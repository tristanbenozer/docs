title: "Audit Analysis Pipeline"
description: "How Hydra structures every audit: repo-specific pre-flight, a deterministic scanner, and six parallel domain agent groups."
---

Every Hydra audit runs two steps before a finding is written. The steps are not configurable — each one is a prerequisite for the next.

## Pre-flight: repo-specific protocol generation

Before any analysis agent runs, Hydra calls a Sonnet model to generate a **repo-specific analysis protocol** for this audit.

The pre-flight call reads:
- Your Discovery profile (conventions, stack, architecture, key paths)
- The repo-specific skill file (if one exists)
- Recent trends for this repo

It outputs a structured protocol that is prepended to every agent's prompt in the next step. This means the six audit agents are not running generic checklists — they are running checklists that have been tuned to your specific codebase before the session begins.

The pre-flight step is always on. It adds a small fixed cost but eliminates false positives that come from generic analysis.

## Step 1: Deterministic scanner

Hydra runs 39 Semgrep-backed scanner patterns against the codebase in parallel. Pattern matches run at zero LLM cost.

Scanner findings are capped at `medium` severity. A finding cannot be `high` or `critical` from pattern matching alone — it must be LLM-verified to reach those severity levels. This prevents false positive scanner patterns from generating high-severity noise.

## Step 2: Six parallel domain agent groups

Six agent groups analyze the codebase simultaneously. Each group owns a distinct domain:

| Group | Focus areas |
|---|---|
| **Security** | Auth, access control, hardening, privacy, compliance |
| **Correctness** | Dead code, naming, structural refactors, AI usage patterns, test coverage |
| **Maintainability** | Coupling, complexity, documentation gaps, dependency hygiene |
| **Performance** | Concurrency, resilience, resource management, cost footprint |
| **Architecture** | API contracts, schema design, database patterns, input validation, multi-tenancy |
| **Operations** | Dependencies, config management, observability, logging, feature flags |

Each agent group:
- Receives the pre-flight protocol
- Receives the relevant global skill file for its domain
- Runs against the full codebase

The six groups run in parallel. Combined output is a deduplicated, prioritized finding pool.

## How deduplication works

Findings are deduplicated using two mechanisms in sequence:

1. **Signature equality** — exact match on finding fingerprint
2. **Embedding similarity** — Voyage embedding cosine similarity at a 0.85 threshold catches near-duplicate findings from different agent groups

Only unique, novel findings ship.
