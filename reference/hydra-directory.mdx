title: "The .hydra/ Directory"
description: "What Hydra writes into your repository, where it writes it, and what each file contains."
---

Hydra writes a `.hydra/` directory into your repository root. Some contents are committed to your main branch (permanent record). Others are generated at runtime and not committed (transient context).

## Committed structure

These files are committed into your repo and persist in version control.

```
.hydra/
├── profile.json              # Discovery output -- your repo's profile
├── conventions.md            # Detected conventions (naming, patterns, stack)
├── skills/
│   └── repo.md               # Repo-specific skill file (editable)
└── fixes/
    └── {ref}/
        ├── finding.json      # Original finding that triggered the fix
        ├── change_plan.md    # The agent's stated change plan
        ├── metadata.json     # Machine-readable fix metadata
        └── accepted_risks.md # Risks acknowledged before the fix ran (if any)
```

## Runtime structure (not committed)

These files are generated during workflow execution and are not committed to your branch.

```
.hydra/
└── workspace/
    ├── audit/
    │   ├── raw_findings.json     # Pre-dedup findings from all agent groups
    │   └── session.json          # Audit run metadata
    └── worktrees/
        └── {ref}/                # Isolated Git worktree for a Fix or Task run
```

## File reference

### profile.json

Written by [Discovery](/workflows/discovery). Contains the full repo profile: detected stack, application type, key entry points, dependency inventory, and the conventions Hydra will apply during analysis.

Do not edit this file manually. Re-run Discovery to regenerate it.

### conventions.md

A human-readable summary of what Discovery detected: naming conventions, file organization patterns, test structure, error handling style. Hydra uses this file to tune agent prompts. You can annotate it with corrections -- Discovery will incorporate them on the next run.

### skills/repo.md

A repo-specific skill file. Each line is an analysis bullet that is appended to agent prompts during every audit of this repo. Hydra writes initial content based on your stack. You can edit this file directly to add codebase-specific guidance.

Example entries:
```
- All API endpoints must validate against a Zod schema before processing
- Use the `createServiceError` factory for all thrown errors in /src/services
- The `legacy/` directory is intentionally not covered by the auth middleware
```

[Learn](/workflows/learn) can also propose additions to this file as `repo_skill` virtual patches.

### fixes/{ref}/

A fix breadcrumb directory. Hydra creates this before any code change runs, commits it into the worktree, and it ships as part of the merged pull request.

The `{ref}` is the finding reference (e.g., `HYDRA-142`).

- `finding.json` -- the full finding schema at the time the fix was triggered
- `change_plan.md` -- what the agent planned to do, written before the first code change
- `metadata.json` -- run ID, model used, timestamp, gauntlet step results
- `accepted_risks.md` -- if the reviewer agent flagged risks that were acknowledged before proceeding, they are recorded here

Every merged Hydra fix is a self-documenting record. You can always look up what Hydra knew and planned before it touched any code.

## Gitignore recommendation

Add the runtime workspace to your `.gitignore`:

```
.hydra/workspace/
```

The committed files (`profile.json`, `conventions.md`, `skills/`, `fixes/`) should be tracked. They are the permanent record and the context Hydra reads on every run.
